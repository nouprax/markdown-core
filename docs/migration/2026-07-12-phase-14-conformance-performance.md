# Phase 14: cross-platform conformance and performance baseline

> Phase 18 supersedes the package-local conformance-fixture portion of this
> snapshot: C, Swift, Kotlin, and ES now enumerate the same root canonical AST
> manifest. Platform routing and the separation from correctness/benchmark are
> unchanged.

## Conformance contract

The `conformance:<platform>` family contains independent native targets that
verify the public parser/binding contract and schema.
It is not included in ordinary
correctness discovery. The command routes directly to native contract
selections:

| Boundary | Native selection |
| --- | --- |
| C facade | CTest `facade_native` and `facade_dump_cli` |
| Swift value tree | Swift Testing `ConformanceSuite` |
| Kotlin value tree | paired Gradle/KMP conformance tasks selecting `AstTest` |
| ES/WASM value tree | package-native `run-conformance.mjs` target |

Phase 15 revised fixture ownership without changing the public schema coverage.
The C adapter exclusively owns and compares the six `.ast` files under
`packages/markdown-core/tests/canonical-ast`. Swift, Kotlin, and ES use focused
package-local inputs to cover all 28 public `Markup` kinds, including
`TableRow` and `TableCell`, behavior-bearing fields, explicit nullability, enums, UTF-8 values,
and start/end scopes. No binding reads C test data, calls the C dump, or consumes
another binding's output. Each binding also owns a focused canonical tree
snapshot generated by its public `TreeDumper` through exhaustive Visitor plus
Walker enter/exit traversal. The diagnostic formatter does not add
property/index edge labels.

## Robustness and benchmark split

Stress describes input pressure, not a public task or suite category. Every
platform's correctness target has `robustness` cases that assert:

- a deterministic document containing 5,000–10,000 heading/paragraph pairs;
- 128 nested block quotes through each copied value tree;
- 2,000–5,000 parse/release cycles with Unicode and extension nodes.

The C facade additionally parses 2,048 nested block quotes. Its scheduled
`deep_nesting` benchmark retains doubling steps at 8,192, 16,384, and 32,768.
During acceptance, Swift's recursive value-copy path passed at 128 levels but
the test process received signal 10 at 256 levels. The portable correctness
level is consequently fixed at 128 for Swift, Kotlin, and ES; this is a
documented boundary, not a skipped test or a parser limit.

The benchmark targets separately time large-document and deep-nesting workloads
with warmup/repeat metadata. Correctness and benchmark may construct the same
deterministic input shape, but they do not share registrations or assertions.
There is no top-level `stress` entry.

## Reproducible baseline workload

The `benchmark:<platform>` family is excluded from correctness and normal
`verify`. Benchmark owns timing, throughput, relative
scaling, and performance baselines; deterministic result/lifecycle assertions
remain in correctness robustness cases. Each binding reports machine-readable
`benchmark` lines. The `large_document` workload uses 2,000 repetitions of one heading and
one paragraph containing strong text, a link, and an emoji (158,000 bytes).
Warmup and repeat counts are printed with every result. The scheduled benchmark
workflow runs C, Kotlin/JNI, and ES/WASM on Ubuntu and Swift in a separate macOS
job, so trends must be compared only within the same runner image and boundary.

Local acceptance snapshot on 2026-07-12 used Apple M-series arm64, macOS 26.5,
Swift 6.3.3, Kotlin/JVM on JDK 26, Node 26.5.0, and Emscripten 6.0.2:

| Boundary | Median | Memory observation |
| --- | ---: | ---: |
| C native parse | 2.404 ms | 7,280 KiB peak RSS |
| Swift native parse + immutable value copy | 3.357 ms | 13,920 KiB peak RSS |
| Kotlin JNI parse + immutable value copy | 10.743 ms | 123,184 KiB RSS; 16,786 KiB used / 65,536 KiB committed JVM heap |
| ES WASM parse + immutable value copy | 7.296 ms | 140,480 KiB peak process RSS |

RSS is process-level and includes each language runtime; Kotlin additionally
prints current RSS plus used/committed heap. These values are trend baselines,
not cross-runtime memory attribution. C reports native parse only; the three
bindings report synchronous parse plus the required full immutable-tree copy.

## Boundary optimization decision

No cross-boundary optimization is introduced in Phase 14. On the common 158 KB
workload every binding completes below 12 ms locally, and the benchmark does
not isolate copying as the dominant cost from parsing or runtime startup.
Changing the direct-accessor copy contract to a serialized bridge would weaken
the ownership and public-surface guarantees without benchmark evidence that
the boundary is the bottleneck. Reconsider only with a representative product
budget, a repeatable same-runner regression, and a profile showing boundary
copying dominates end-to-end parse time or retained memory.
