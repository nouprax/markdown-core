# Markdown Core 2.0.0

This release adds incremental parsing sessions to every platform. A session
owns one Markdown text and its living AST: callers apply byte-range edits
(streamed appends are edits at end-of-text) and commit, and the engine
reparses only the stale region around the edits at a per-commit cost that is
independent of document size. One-shot `Document.parse` is unchanged and
keeps its previous memory profile.

- `MarkupSession` in Swift, Kotlin, and ECMAScript and
  `markdown_core_session_*` in C provide edit, commit, snapshot, and
  node-lookup operations over one living document.
- Every commit yields an immutable document snapshot that structurally
  shares unchanged nodes with the previous snapshot, plus a `Delta`: the
  exact set of node ids the commit added, removed, or changed, with
  before/after revisions.
- Node identity is stable across commits. A `MarkupID` carries the session
  lineage and keeps addressing the same node until it is removed; ids
  resolve against the latest snapshot on every platform.
- After any sequence of edits and commits, the session's document is
  canonically dump-equal to a from-scratch parse of the same text. This
  equivalence is enforced by replay suites over the shared conformance
  fixtures, the CommonMark specification corpus, seeded random edit
  scripts, and a coverage-guided edit-script fuzzer.
- Sessions answer footnote queries directly — numbering, resolution state,
  first-use order, and back-references — and keep them incremental: footnote
  index refreshes are bounded by the affected sites, not the document.
- Commits are transactional: a failed commit leaves the session valid at its
  previous revision and retryable, with applied edits retained.
- There is no process-global parser state. Any number of sessions can run
  concurrently; Swift snapshots are `Sendable`, Kotlin sessions work on JVM
  executors and native workers, and each WebAssembly worker instantiates
  independently.
- The incremental cost model, identity and ordering rules, and delta
  semantics are specified in `docs/specs/sessions-and-deltas.md` as a
  language-neutral contract binding all platform APIs.

The major version reflects the scale of the addition and coordinated
platform-surface changes around it, including document-mediated dump and
walk entry points and session-aware node values. Adversarial-input coverage
(inline floods, kilodeep block chains, reference-map churn, restart-hostile
documents, CRLF seams) and allocation-failure injection run against the
session machinery in CI alongside the existing sanitizer and conformance
gates.
